Roadmap

Features planed for QACZAR.

  • Consider a debug only function that also stores the message in a log file.
  • Test this with multiple requests that write to the database.
  • Define the columns based on the function signature automatically.
  • Don't mix this logic with FORMS, recorded can be used for more.
  • Automate CSS classes and data attributes (htmx)
  • Generate the CSS code dynamically instead of reading a file.
  • Consider tracking components with the database instead of a global.
  • Consider additional attributes for the header / nav.
  • Each tenant should have their own SSL certificates.
  • Add more parameters to the access log.
  • Performance testing is needed to ensure this approach will work in the long run.
  • This creates 1 thread per request, which is not ideal. Implement a thread pool.
  • Design a string for the user-agent and use it to track tests.
  • Test other special paths such as blank, /, etc.
  • Test submitting a form (ie. sign_guestbook).
  • Create missing branch if not exists when pushing to git.
  • Catch port being in use and suggest a different port.
  • Add automatic tests to prevent public API regressions.
  • Keep the tester running and re-run tests when source changes.
  • Have a keep-alive ping to detect when the server is down.
